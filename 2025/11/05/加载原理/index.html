加载原理

更新于2025-11-05|内存操作
|浏览量:

https://www.cnblogs.com/theseventhson/p/14732937.html（参照此文章，非常好，作者太棒了）

  1. Windows如何在点击exe文件后即可运行？

    • 为啥能运行
    • 运行的本质是什么
    • 为什么运行的exe进程可以被explorer发现
    • 如何让进程不出现在explorer中

    在windows下，可执行文件必须满足一定的格式，微软官方称之为PE文件（关于PE文件的详细介绍在文末），用户在双击exe，有个叫explorer的进程会监测并接受到这个事件，然后根据注册表中的信息获取文件名，再通过explorer进程的createprocess函数去运行用户点击的exe，所以进程管理器中很多进程都是explorer的子进程


    发现有不少进程不是explorer的子进程，这些进程在启动时，会询问以管理员权限启动

    explorer形式启动主干流程：

    • 分配内存：首先调用explorer进程的createprocess函数，在内存为新进程分配内存，生成页表、填充CR3等工作

    • 加载到内存：分配内存之后，将exe文件加载到内存

    • 重定位：PE文件在被编译器编译的时候，编译器不知道文件会被加载到哪个VA（一般exe文件默认从40000开始，dll文件从100000开始），一般一个exe会加载多个dll文件，后面加载的dll肯定会和前面加载的dll的文件的imagebase冲突，这个时候需要把dll或者exe文件加载到其他地址，一旦修改了地址，全局变量/局部变量，函数调用等硬编码的地址都要更改；所以PE文件单独有个relc段，表明了需要重新定位和生成VA的地址；由于硬编码放的都是相对地址，重定向后的VA的计算公式入下：新imagebase-旧imagebase+RAV（概念问题在文末解释）

    • 填写导入表：当我们调用系统函数的时，explorer需要在导入表中找到对应的系统调用函数的地址，用此地址替换PE文件中的callfuntion的地址（这也是为什么OD,x64dbg在调试时能找到这些系统函数的原因），导入表位于exe文件中，编译器和连链接器在生成exe文件时，根据源代码中对外部函数的调用生成导入表，用于explorer填写。（填写时系统函数的地址可以直接填写，dll中的函数需要把dll完全导致之后获取函数地址并填写导入表）

    这就是进程被加载的主干过程，所有被系统自动加载的进程，包括普通进程何管理员进程（不属于explorer子进程）均会被windwos获取信息，若是想实现不被察觉，需要自己实现exe或dll的加载和运行

    说明1：

    （1）imagebase：整个文件（PE，sys，dll）在虚拟内存中的起始地址，上面说的重定位也是从imagebase这里开始重新计算地址

    （2）VA：在虚拟内存中的地址

    （3）RVA：与当前区段的相对地址（.text，.data段等）

    （4）PointerToRawData：段头部地址在源磁盘文件中的相对于文件开头的距离

    （5）FOA: file offset address 文件内偏移地址 = PointerToRawData+RVA找到虚拟地址与源磁盘文件位置之间的关系
